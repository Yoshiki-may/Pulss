0. 前提・スコープ

対象範囲

営業〜ディレクション〜運用〜効果測定までのワークフロー。

その中で語られている「AI（GPT等）の役割・効率化ポイント」のみを対象。

ダッシュボードのUI/UXは別設計。ここでは「UIから呼ばれるAIバックエンドAPI」のみ定義。

モデル構成の前提

高性能な有料LLM（例: GPT系など）と、社内スーパーコンピュータ上の無料LLMを併用する二段構成。

用途と必要精度に応じて「標準（低コスト）」「高精度（高コスト）」モデルを選べるようにする方針。

社内ナレッジ（案件実績・リファレンス動画・効果データなど）をAIに学習させ、パルス専用の“法人AI”として機能させる。

タスク自動化の方針

AIが勝手にタスクを発行・完了させるのではなく、「AIは議事録や候補タスクを提示し、人間が取捨選択する」設計。

1. 全体ワークフローとAI配置イメージ

営業準備・事前リサーチ

初回商談・録音/議事録 → 宿題AIフォーム

宿題回答から「クライアント背景レポート」生成

提案資料・企画案・台本の自動生成

受注後キックオフ → 企画会議サポート

運用中のチャット/ミーティングの要約・タスク候補化

効果測定・成果レポート・次月提案の自動生成

営業/現場の会話ログを使った教育・品質向上

それぞれに対応するAIバックエンド機能を定義していきます。

2. 章別 AI機能要件定義
2-1. 営業準備・事前リサーチ
AI-01 企業自動リサーチAPI

目的

営業が商談前に行っている「企業情報リサーチ」を自動化し、短時間で深い事前準備を可能にする。

ホームページ・求人・ニュース・SNSなどから、現状・課題・トーンをサマリ。

入力

{
  "client_id": "string",
  "company_name": "string",
  "domain": "string | null",
  "industry": "string | null",
  "preferred_language": "ja"
}


処理（LLMまわり）

Webクローラ/外部APIで以下を収集（非AI部分）：

コーポレートサイト、採用ページ、プレスリリース

公開SNSアカウント（Instagram/TikTok/YouTube等）

可能なら求人媒体（Wantedlyなど）の求人情報

収集テキストをRAG（ベクトル検索）で整理し、LLMに渡す。

LLMは以下のセクション構造で要約：

企業概要

事業・サービス

ターゲット層・ブランドイメージ推定

既存SNS/マーケ施策の整理

想定される課題仮説（※事実と推測を分けて出力）

出力

{
  "summary": "営業が1〜2分で読める自然文",
  "sections": {
    "company_profile": "...",
    "business_overview": "...",
    "brand_and_tone": "...",
    "existing_marketing": "...",
    "hypothesized_issues": "..."
  },
  "source_documents": [
    {"type": "url", "url": "...", "title": "..."}
  ]
}


優先度: P1（宿題レポートのほうが優先 / 後で連結）

2-2. 要件ヒアリング（宿題AIボット）
AI-02 要件定義ボットAPI（宿題AI）

目的

「AIがどんどん質問してユーザーの要望をちゃんとヒアリングしてくれるボット」を実装し、
営業前/営業後の宿題フォームとして使えるようにする。

従来の「書くのが面倒」「背景をちゃんと書けていない問題」を解消。

利用シナリオ

パターンA: テレアポ後〜初回商談前にURLを送付し、事前ヒアリング。

パターンB: 初回商談後に「宿題」としてURL送付し、詳細を記入してもらう。

入力

{
  "client_id": "string",
  "project_id": "string | null",
  "locale": "ja",
  "mode": "pre_meeting | post_meeting",
  "conversation_history": [
    {"role": "user", "content": "過去のやりとりなど", "timestamp": "..."}
  ]
}


対話設計（LLM）

システムプロンプトで以下を固定ルールとして定義：

5分程度で完了する質問数に抑えるが、重要な前提は落とさない。

企業規模・業種・エリア・店舗数

SNS導入の目的（採用 / 集客 / ブランディングなど）

KGI/KPI（例: 応募数・サイト流入数・売上・予約数 etc.）

既存の課題・現状（人手不足/工数不足/クレームリスクなど）

一番こだわっているポイント・押したいポイントの確認。

回答ごとにJSON構造にマッピングしながら会話を進める（slot-filling）。

出力

{
  "conversation_id": "string",
  "completed": true,
  "raw_qa": [
    {"question": "string", "answer": "string"}
  ],
  "structured_requirements": {
    "company": {...},
    "objectives": {...},
    "constraints": {...},
    "preferences": {...}
  }
}


API設計例

POST /ai/requirements/dialog/start

POST /ai/requirements/dialog/answer

GET /ai/requirements/dialog/{conversation_id}/result

優先度: P0（最優先）

2-3. 宿題回答 → クライアント背景レポート生成
AI-03 クライアント背景レポート生成API

目的

宿題AIと営業の商談内容、録画議事録から「現状の整理 + ゴール + こだわりポイント」が整理されたレポートを自動生成。

ディレクション側がレポートを見れば「なぜこの企業がSNSをお願いしたか」が一目で分かる状態にする。

入力

{
  "client_id": "string",
  "project_id": "string",
  "requirements_json": {...},          // AI-02 の structured_requirements
  "meeting_transcripts": [             // 営業商談の文字起こし
    {"meeting_id": "string", "text": "string", "timestamp": "..."}
  ],
  "research_summary": {...}            // AI-01 の結果（任意）
}


処理

RAGで宿題回答・議事録・事前リサーチを統合。

LLMに以下のテンプレ構造で生成させる：

企業概要

現状の整理（課題・背景）

SNS導入の目的（採用・集客・ブランド etc.）

KGI / KPI定義

一番こだわりたいポイント / 押し出したいメッセージ

想定リスク・前提条件

出力

{
  "client_brief_id": "string",
  "summary_markdown": "### 現状の整理 ...",
  "sections": {...},  // 上記構造
  "source_links": [...]
}


優先度: P0

2-4. 提案資料・企画・台本生成
AI-04 提案骨子生成API

目的

営業が毎回2〜4時間かけて作っている提案資料の「骨子・文章部分」を自動生成し、作業時間を大幅に削減。

入力

{
  "client_brief_id": "string",
  "pricing_patterns": [
    {"plan_id": "A", "budget": 300000, "description": "..."},
    {"plan_id": "B", "budget": 500000, "description": "..."}
  ],
  "internal_templates": {
    "proposal_outline": "現状→ゴール→プランABC→期待効果→工数→まとめ"
  }
}


処理

LLMにテンプレ構造を渡し、「現状の背景→ゴール→プランABC→期待効果→工数→まとめ」という流れで埋めさせる。

プランごとの差異（役務の違い・工数の違い・リスクの違い）を明確に説明させる。

出力

{
  "proposal_outline_md": "## 現状の整理 ...",
  "plan_details": [
    {"plan_id": "A", "description_md": "..."},
    {"plan_id": "B", "description_md": "..."}
  ]
}

AI-05 企画・台本生成API（感性モデル利用）

目的

「企画の部分が一番しんどくて、今はやれていない」部分を、感性強めのモデルで補完する。

Open 4.5 / Claude など「文章表現に強い外部モデル」を使って企画案・撮影台本を生成する。

入力

{
  "client_brief_id": "string",
  "proposal_outline": {...},            // AI-04 の結果
  "reference_assets": [                 // リファレンス動画等
    {"id": "ref1", "url": "...", "tags": ["採用", "エモい", "ホテル"] }
  ],
  "tone": "serious | casual | pop | luxury"
}


処理

感性モデル（外部LLM）に対し、以下を指示：

提案資料の「企画」セクション用の要約テキスト

1〜3本の動画コンセプト

各コンセプトの簡易台本（構成・セリフ案・カット割り）

参考リファレンス動画をプロンプトに組み込みつつ、社内で蓄積した「良い企画パターン」のナレッジもRAGで参照。

出力

{
  "concepts": [
    {
      "title": "採用ストーリー動画案A",
      "summary": "...",
      "script_outline": "...",
      "use_references": ["ref1","ref3"]
    }
  ]
}


優先度: P1（背景レポート → 骨子生成の後）

AI-06 提案資料ファイル生成API

目的

AI-04/05で生成した文章をもとに、PowerPoint / Google Slides などの初稿ファイルを自動生成。

営業は「チェックと微修正」に集中できる状態にする。

入力

{
  "project_id": "string",
  "proposal_outline": {...},
  "visual_template_id": "string"
}


処理

LLMは使わず、テンプレートとテキストをマージするドキュメント生成処理（非LLM）。

ただし「タイトルの言い回し」「キャッチコピー」はLLMで微調整可能にしても良い。

出力

{
  "file_url": "s3://.../proposal-v1.pptx"
}


優先度: P2（なくても回るが、あると爆速）

2-5. キックオフ〜企画会議支援
AI-07 キックオフ議事録 & 要件テンプレ自動反映API

目的

キックオフMTGを録画/録音し、議事録＋タスク候補＋テンプレートへの自動反映を行う。

入力

{
  "meeting_id": "string",
  "audio_url": "string",
  "client_id": "string",
  "project_id": "string",
  "template_schema": {...}    // 「社内テンプレ」項目定義
}


処理

音声→文字起こし（既存AI文字起こしツール）

LLMにより：

議事録（要約＋詳細ログ）

決定事項・未決事項

社内テンプレの各項目に自動マッピング

タスク候補一覧を抽出（誰が / 何を / いつまで）

出力

{
  "minutes_md": "...",
  "decisions": [...],
  "open_issues": [...],
  "template_filled": {...},
  "task_candidates": [
    {
      "task_text": "採用用リールの構成案を3本出す",
      "assignee_suggestion": "director",
      "due_date_suggestion": "2025-12-20"
    }
  ]
}


タスクはあくまで「候補」。ユーザーがUI上で選択して正式タスク化する設計とする。

優先度: P1

AI-08 企画会議サポート / リファレンス検索API

目的

企画会議で「ヒロさんの感性」など属人的なリファレンス共有を、AIがサポート。

社内リファレンス動画 + 世界中のリファレンスをAIが引っ張ってくる「ディープリサーチ機能」。

入力

{
  "client_brief_id": "string",
  "keywords": ["採用", "ホテル", "エモい"],
  "platform_preference": ["instagram", "tiktok", "youtube"],
  "max_results": 20
}


処理

社内リファレンスDB（動画URL＋タグ＋感想コメント）をベクトル検索。

外部SNS/動画プラットフォームAPI検索結果も取得。

LLMで「なぜこのリファレンスが合うのか」の説明コメントを生成。

出力

{
  "references": [
    {
      "source": "internal",
      "url": "...",
      "reason": "同じくホテル採用で、社員インタビュー主体の構成"
    },
    {
      "source": "external",
      "platform": "youtube",
      "url": "...",
      "reason": "ブランディングトーンが近い"
    }
  ]
}


優先度: P2

2-6. 運用中チャット・ミーティングの要約とタスク候補化
AI-09 チャット集約・文脈理解API

目的

クライアントごとにバラバラなチャットツール（Slack, Teams, Chatwork, LINE公式など）の履歴をAIが横断的に理解し、状況把握と提案の材料に使えるようにする。

将来的には「チャット履歴＋レポートから1ボタンでAI提案」を実現。

入力

{
  "client_id": "string",
  "channels": [
    {"type": "slack", "channel_id": "..."},
    {"type": "teams", "channel_id": "..."},
    {"type": "line_official", "room_id": "..."}
  ],
  "time_range": {
    "from": "2025-12-01T00:00:00Z",
    "to": "2025-12-09T23:59:59Z"
  }
}


処理

各チャットツールのメッセージを収集（コネクタ層）。

LLMで以下を抽出：

重要な要望・クレーム・褒め言葉

未対応の質問

依頼内容と対応状況

文脈を保持したサマリを生成。

出力

{
  "summary_md": "### 直近1週間のトピック ...",
  "open_requests": [...],
  "risk_signals": [...],
  "raw_threads": [...]
}


優先度: P2（チャット統合自体が後フェーズ）

目的

3. 共通AI基盤の機能要件
3-1. モデルレイヤ / コスト制御

要件

「無料社内LLM」と「高性能外部LLM」を用途に応じて選択できる仕組み。

各APIに model_tier パラメータを持たせる：

{
  "model_tier": "standard | premium"
}


推定コスト例（会話内で言及）:

標準: 1回あたり ~50円（宿題レポートなど）

高精度: ~500円（大型案件・重要レポートなど）

ダッシュボード側で、案件単位・クライアント単位で使用上限を設定できるようにする（ここは設定API側の要件）。

3-2. ナレッジベース

要件

下記を継続的に取り込むベクトルDB（RAG基盤）：

宿題AIの構造化要件

提案資料・効果測定レポート

チャット履歴（機微情報をマスキングした形で）

リファレンス動画＋タグ＋評コメント

幹部会や現場会議の議事録

クライアントごとにナレッジを分離し、NDAに配慮（案件間でデータを混ぜない）。

要件

NDA契約範囲外のデータは学習・再利用禁止（モデルのファインチューニングに使わないなど）。
