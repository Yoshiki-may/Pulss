## 1. 現状の課題

### 1-1. タスク・進捗管理（ディレクター業務）

- タスク管理が
    
    - **Slackの日報チャンネル（今日やるタスク報告）**と
        
    - **案件ごとのSlackチャンネル（連絡・報連相）**  
        に分散しており、
        
    - 過去タスクが**埋もれやすい**
        
    - 各案件の**全体進捗が直感的に把握しづらい**
        
- 1人のディレクターが**最大10クライアント弱**を担当し、頭の中とSlackで管理しているため、
    
    - 抜け漏れリスクが高い
        
    - 「どの案件で何が今足りていないか」が一目で分からない
        
- 新規案件発生時に
    
    - 制作スケジュール作成
        
    - アカウント情報・権限付与
        
    - ロゴ・画像・素材ドライブの手配  
        などの**初期タスクが大量発生するが、標準化されておらず人依存**。
        
- クライアントからの素材・ロゴ待ちタスクなど  
    「○日までに連絡しておかないと、後工程が詰む」系のタスクが  
    **各自の勘と経験に依存している**。
    

### 1-2. コミュニケーション／危機管理・顧客視点

- ディレクター陣は
    
    - 「目の前のタスク処理」で手一杯になりがちで
        
    - クライアント側の心理・将来のリスクを見越した**危機管理・先回りコミュニケーションが弱い**。
        
- 撮影前の事前連絡や、素材遅延時のスケジュール確認など  
    「今のままだとクライアント体験的に危ないよね？」という視点での  
    **注意喚起が山ちゃんに集中**している。
    
- 山ちゃんが
    
    - 日中は打ち合わせで手一杯
        
    - 夜にまとめてLINEや連絡を確認しているが、  
        そこで気づいたことを**即座にメンバーへ安全な形でタスク化できない**（深夜に連絡しづらい）。
        

### 1-3. SNS運用・企画の質と属人性

- 運用開始後、企画のレイヤーが
    
    - 「現状維持」レベルに留まりがちで
        
    - 初回撮影時に作った企画を**そのまま惰性的に回し続ける傾向**。
        
- 「2ヶ月分撮影 → 2ヶ月分投稿」はできている一方で
    
    - 残り1ヶ月時点での**次企画提案（トレンド反映・新しい構成提案）**がディレクターには難しい。
        
- 高度なマーケティング視点・コンセプトメイクは
    
    - ほぼ**山ちゃん一人に集中**しており、
        
    - 若者インサイト・市場データから戦略〜コンセプト〜キーコピーまで落とし込むプロセスが**属人化**。
        
- クリエイティブ側（ひろにぃ）は
    
    - 世界観・センスで海外リファレンスを探し出しているが
        
    - 1案件あたり複数本分のリファレンスを毎回**手作業検索しているため負荷が高い**。
        
- プロジェクトによって
    
    - 「ゴリゴリ戦略から提案してほしい案件」と
        
    - 「おもしろいアイデアだけ出してほしい案件」  
        が混在しているが、**ワークフローとしては統一されていない**。
        

### 1-4. 要件定義・ヒアリングの課題

- 新規案件のヒアリングは主に**あなんちゃん（営業）**が行い、  
    山ちゃんは同席しないケースが多い。  
    → クライアントの本当の課題・背景が**十分に構造化されていない状態でパスされる**。
    
- 多くのクライアントは
    
    - 「とりあえずSNSやった方が良さそうだから」レベルの認識で、
        
    - 自社の本質的な課題やゴールが**言語化されていない**。
        
- そのため、
    
    - 案件ごとに山ちゃんが**ゼロから整理→提案資料を作る**必要があり、
        
    - 負荷・ボトルネックになっている。
        

### 1-5. AI活用のばらつき・ナレッジ蓄積の不足

- 山ちゃんは
    
    - ChatGPT Pro / Deep Research を駆使して
        
    - 調査 → データ整理 →インサイト抽出 → 提案資料作成  
        を一人でやっているが、  
        他ディレクターは**プロンプトを扱えない/扱いにくい**。
        
- クライアントごとにChatGPTのチャットを**毎回新規で作成しているため**、
    
    - 過去案件のナレッジが横断的に蓄積されず、
        
    - 「PULSSらしい思考」をAIに継承できていない。
        
- 過去の提案資料・レポート・結果データが  
    **一元的にAI学習用のラグ（RAG）として整理されていない**。
    

### 1-6. リサーチ・資料作成の工数

- 以前は
    
    - 1案件あたり**1週間近くリサーチに費やす**こともあり、
        
    - 今もDeep Researchを使いつつ、山ちゃんが中心となり  
        大量の調査→整理→資料化を行っている。
        
- この工数が大きく、  
    **山ちゃんの稼働が常にボトルネック**になっている。
    

### 1-7. 人員アサイン・組織運用（今後深掘り予定）

- 案件ごとに
    
    - どのディレクター
        
    - どのクリエイター（ひろにぃ含む）  
        を付けるかは、現状**山ちゃんの頭の中の感覚値**で判断されている。
        
- 各メンバーの
    
    - トンマナ
        
    - 思想・強み
        
    - レベル感  
        と案件特性のマッチングが**仕組み化されていない**。

## 2. 解決していくこと（システム／AIで実現したい方向性）

### 2-1. ディレクターボード & タスク可視化

- **案件ごとのタスク標準化 & 初期セット自動生成**
    
    - 契約完了時に
        
        - アカウント情報取得
            
        - バイパス/ログイン権限付与
            
        - ロゴ・素材の受領
            
        - ドライブリンク共有
            
        - 初期制作スケジュール作成  
            などの**「初期チェックリスト／ガント」**を自動生成。
            
    - 期限設定（例：1週間以内完了）と**カレンダー/リスト表示**。
        
- **ステータス管理（対応状況・進捗確認）**
    
    - デモ画面の  
        「ヒアリング → 提案作成 → 見積もり → 制作 → 制約 → 契約 → キックオフ → 運用中」  
        のようなフェーズを**見える化**。
        
    - フェーズ完了までは「導入タブ」、完了後は「運用中タブ」に切り替え、  
        運用フェーズ専用のタスク画面で細かい運用タスクを管理。
        
- **ディレクターボード（1元管理）**
    
    - 山ちゃん視点で
        
        - 全クライアントの進捗・リスク状態を一覧で見られるボード。
            
    - そこから各案件のディレクターへ
        
        - 「クライアントにスケジュール確認依頼」などの**タスクをワンクリック追加**  
            → 翌営業日などに通知。
            
- **通知・アラート**
    
    - 初期タスクが期日までに完了していない場合
        
        - 色付け（赤マークなど）や
            
        - リマインド通知（3日目・7日目など）を自動送信。
            
    - 「しばらくクライアントと連絡を取っていない」など、  
        **危険な状態を自動検知し通知**。
        

### 2-2. クライアント対応の先回り・自動提案

- システム上に
    
    - 初期要件（ゴール・ターゲット・世界観）
        
    - 投稿/撮影履歴
        
    - 各コンテンツへのクライアントコメント
        
    - 数値結果  
        を蓄積し、AIがこれらを元に
        
    - 「次回撮影前に送るべき提案文面」
        
    - 「トレンドを踏まえた新企画案」  
        を**自動生成**。
        
- 例：
    
    - 「次の撮影日が近づいたので、前回のトレンド＋クライアントのコメントを踏まえた企画案と共に事前連絡を提案」
        
    - 「コンタクトが◯日空いているクライアントに、新しい切り口の連絡案を提案」
        
- まずは**60％程度の精度の叩き台**として出し、
    
    - 山ちゃんにエスカレーション → フィードバック
        
    - 精度を高めて最終的には**90％前後を目指す**。
        

### 2-3. 要求定義ボット「パルスくん」（クライアント用チャットボット）

- クライアントの
    
    - 業種
        
    - プロダクト
        
    - USP（強み）
        
    - 既存顧客の属性（年齢層など）
        
    - 目標（ブランディング／認知拡大／採用／売上…）
        
    - 現状の課題感  
        を**会話形式で深掘り**するボットを作成。
        
- さらに可能な範囲で
    
    - 「なぜその強みが刺さると思うか」
        
    - 「なぜ顧客は自社を選ぶと思うか」  
        といった**クライアントの仮説・本音**も収集。  
        → ただし、ここは**必須ではなく任意／補助情報**として扱う。
        
- 出力は
    
    - テンプレート化された**要件定義レポート**（要件定義書の下地）として整形し、
        
    - システムでそのまま参照・AIが利用できる状態にする。
        
- これにより
    
    - あなんちゃん→山ちゃんへのパスの前段で、
        
    - クライアントの要求が**深く構造化された状態**にできる。
        

### 2-4. PULSS専用AI（パルスくんの「頭脳」）の構築

- 内部ナレッジをAIに統合：
    
    - 過去の提案資料
        
    - 戦略・コンセプトの作り方（例：アートの採用資料）
        
    - 実績・数値結果
        
    - 山ちゃん／ひろにぃ／他メンバーのノウハウ  
        をRAG（ラグ）として読み込ませる。
        
- それを基にAIが：
    
    - 戦略の骨子（ゴール → 課題 → ユーザーインサイト → コンセプト）
        
    - コンテンツ企画案
        
    - クリエイティブの方向性の候補  
        を**PULSSらしい思想で生成**。
        
- 出力に対して
    
    - 「良い／悪い」「ここを修正」などのフィードバックを蓄積し、
        
    - 半年程度かけて**強化学習的に精度向上**を狙う。
        

### 2-5. リファレンス抽出・台本生成の自動化

- 山ちゃんのコンセプトメイク情報と  
    ひろにぃのリファレンスの傾向（海外中心の世界観）を学習させ、
    
    - 「このコンセプトなら、こういう世界観のリファレンス」
        
    - 「短尺ドラマなら、このようなエピソード構成・モノローグ案」  
        をAIが自動生成。
        
- 例：
    
    - 「サラリーマンブルースコード」のような企画に対して
        
        - パワハラ気味の部長の心情モノローグ＋ブルースコードのメロディ、  
            といった**具体的な1話分の台本イメージを自動で複数パターン生成**。
            
- 納得いくまで**ボタン一つで再生成**できるようにし、  
    ひろにぃの「ゼロから探す負荷」を大幅に軽減。
    

### 2-6. リサーチ・資料化エージェント

- 朝ボタンを押すと夕方までに：
    
    - 市場リサーチ
        
    - 競合調査
        
    - ユーザーインサイト候補の抽出
        
    - スライド案のドラフト  
        までを自動で行う**調査エージェント**を構築。
        
- 人は最後の**2〜3割のブラッシュアップ・意思決定**に集中できる状態を目指す。
    

### 2-7. 人員アサイン機能（今後詳細設計）

- メンバーごとに
    
    - 得意業界
        
    - トンマナ・思想
        
    - スキルレベル  
        をデータ化し、
        
    - 案件特性（目的・業界・予算など）と照合して  
        **最適なディレクター／クリエイターの組み合わせを提案**する機能。
        
- 山ちゃんがダッシュボードから
    
    - プランA/B/Cのようなアサイン案や企画の叩き台をポチポチ選んで
        
    - 関係者にそのまま引き継げるフローを想定。

## 3. 既に決まっていること・合意された方針

1. **この文字起こしを要件定義書に落とし込み、エンジニアに渡す方針は確定。**
    
    - 本打ち合わせは、そのためのディレクション情報の洗い出し。
        
2. **ディレクターのタスク・進捗管理を「見える化するボード」で一元管理する方針。**
    
    - 日々のタスク報告・案件別連絡を  
        Slackだけではなく、**専用ダッシュボードで管理する方向性**に合意。
        
    - フェーズ管理（ヒアリング〜提案〜契約〜キックオフ〜運用中）を  
        システムで持つこと自体は**YES**。
        
3. **新規案件の「初期タスクチェックリスト」をテンプレ化し、自動生成する。**
    
    - アカウント情報、素材受領、スケジュール作成など  
        **最初の1週間で終わらせるべきタスク**が一覧化される設計にする。
        
    - 期限超過時の**アラート表示・通知**を入れる方向で合意。
        
4. **運用フェーズ用の別タブ／画面を用意する。**
    
    - 商談〜導入フェーズのタスクと  
        運用中の細かいタスクは**UI上で分ける**方針。
        
    - 運用中タブでは、山ちゃんが気づいたことを  
        各案件にタスクとして追加 → ディレクターに通知する仕様を目指す。
        
5. **クライアント要求定義ボット（パルスくん）をまずGPTsで試作する。**
    
    - 土日で開発者側が**GPTsでボットを作成することを約束**。
        
    - 日曜日に山ちゃんが
        
        - 仮想企業として回答 →
            
        - 出力されるレポートの妥当性を一緒に確認する、という**次のアクションが決定**。
            
    - ここで得られたフィードバックをもとに  
        要求定義テンプレート＆質問設計をブラッシュアップしていく。
        
6. **ディレクターには基本的に「プロンプトを書かせない」前提で設計する。**
    
    - AIは**裏側に組み込むエージェント**として動き、
        
    - ディレクターはボタンや簡単な入力だけで  
        タスク生成・提案文面・企画案などを得られる設計にする。
        
7. **山ちゃん自身は引き続きAI（ChatGPTなど）をフル活用し、AI側の教育役を担う。**
    
    - 既に行っている
        
        - 提案資料へのフィードバック
            
        - 「クライアント目線でどこが引っかかるか」チェック  
            を、今後は**PULSS専用AIの学習にも活かしていく**方針。
            
    - 「人の言葉で話すべき部分」と「AIが提案してよい部分」の線引きも  
        今後詳細に決めていくことで合意。
        
8. **次回（日曜）の打ち合わせで深掘りするテーマが確定。**
    
    - 要求定義ボットの精度確認。
        
    - 案件ごとの**ディレクター／クリエイターのアサインロジック（トンマナ・レベル感含む）**。
        
    - それをダッシュボードでどう扱うかの方向性。
